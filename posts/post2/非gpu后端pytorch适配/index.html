前言

本文主要分析非GPU后端如TPU这种专用神经网络处理器对于Pytorch的适配，主要分析对象是谷歌的Pytorch/XLA，寒武纪以及GraphCore对于Pytorch的适配。
为了打败GPU或者说提升神经网络的训练推理效率，各种非GPU架构芯片近年来层出不穷，比较出名的有谷歌的TPU，寒武纪的NPU，百度的XPU，华为从寒武纪那里继承来的NPU。通常这些*PU都无法直接运行Pytorch，因为Pytorch的动态图对于硬件的要求还是非常高的。所以也就出现了各种Pytorch Adapter。也就是本文的分析对象。
网上能够获取到的开源Pytorch Adapter如下:

  • Pytorch/XLA
  • 寒武纪(Cambricon PyTorch)
  • GraphCore(PopTorch)

那么接下来我们就从原生pytorch开始分析吧～

raw Pytorch

Pytorch在NIPS会议上发表了《PyTorch: An Imperative Style, High-Performance Deep Learning Library》论文，论文详细阐述了Pytorch的设计思路。

论文概要

设计理念

文中总结了四点Pytorch的设计理念

  • Pythonic
  • Put researchers firs
  • Provide pragmatic performance
  • Worse is better

第四点主要是内涵了tf一波，与其提供一个功能完备，但是无比复杂的框架，Pytorch认为简单的实现更能够让研究者们把更多的精力去跟上AI技术的飞速发展。Worse is better的哲学是Simplicity，Correctness，Consistency，Completeness，有兴趣的读者可以阅读Worse is better

Pytorch利用与Numpy数据的互相转换来作为与各种第三方库的桥梁，因为Numpy可以说是Python计算的基石。同时Pytorch也利用了重载运算符的方式实现了自动微分。后续会详细一篇文章解析Pytorch的自动微分实现。

设计概要

An efficient C++ core

Pytorch实现了一个非常高效的C++内核。这里集合了Tensor这种基础数据结构的实现，各种设备的算子实现，以及内存管理实现等等，主要是集中在C10（Caffe Tensor Library），ATEN（A tensor Library）中。

Separate control and data flow

Pytorch将数据流与控制流（if、for、switch）分离，利用了CUDA的stream机制，可以异步下发Kernel，通常情况下，device的执行时间是要高于host的下发时间的，所以这种机制可以让device一直处于计算过程中。但控制流是在host上执行，device运算结束后，其值返回host，并在host上解析。

Custom caching tensor allocator

Pytorch实现了对与Device的内存管理，pytorch/c10/cuda/CUDACachingAllocator.h就是对于GPU的内存管理实现。Pytorch为每个stream维护一个内存池。

Multiprocessing

主要讲述Pytorch如何解决多进程场景下Python的GIL锁问题

Reference counting

引用基数问题，Pytorch将Deivece内存视为稀缺资源，所以Pytorch中多个Tensor其实是共享同一块内存，只是其View(视图)不同。

Pytorch的每个设计概要都需要数篇文章去文章去详细解析。后续会慢慢补上。 这篇文章可以作为一个初学者的提纲，也是记录我对Pytorch的学习过程。只有将上述五个设计点解析清楚，我们才能够去解析各家的适配，以及他们为什么选择这么适配。
