前言

本文将解析Pytorch C++ core中的tensor实现

intrusive_ptr

c10/util/intrusive_ptr.h
首先先介绍的是intrusive_ptr，一种侵入式的指针。 首先先看这种场景

int *a = new int(5);
std::shared_ptr ptr1(a);
std::shared_ptr ptr2(a);

当你利用同一个裸指针去初始化两个智能指针时，在这两个智能指针被析构时，这个裸指针就被double free了。 那么我们来看intrusive_ptr如何解决这个问题。

class C10_API intrusive_ptr_target {
  mutable std::atomic<size_t> refcount_;
  mutable std::atomic<size_t> weakcount_;

  template <typename T, typename NullType>
  friend class intrusive_ptr;
  friend inline void raw::intrusive_ptr::incref(intrusive_ptr_target* self);

  template <typename T, typename NullType>
  friend class weak_intrusive_ptr;
  friend inline void raw::weak_intrusive_ptr::incref(
      intrusive_ptr_target* self);
}

首先定义一个intrusive_ptr_target类，假设T类型的class需要被intrusive_ptr管理，那么首先他需要继承这个类，即

class T : public intrusive_ptr_target{}

那么该类T就继承了父类的两个原子类型的成员变量refcount_与weakcount_。分别表示该类T的实例被多少个intrusive_ptr指向和多少个weak_intrusive_ptr指向。

由于这两个成员变量是私有成员变量，那么为了让intrusive_ptr和weak_intrusive_ptr能够访问这两个变量，这两个类以及这两个增加引用数的函数被申明乘友元类以及友元函数。

auto in_ptr = intrusive_ptr(TTarget* target)

假设我们调用了这个一条语句，先不管正确性，通过构造函数，in_ptr保存了一个TTarget*类型的裸指针，并且他可以去获取裸指针继承自intrusive_ptr_target而得来的两个成员变量，refcount_与weakcount_。在intrusive_ptr的拷贝构造函数中，他需要去将 TTarget* target的refcount_加1，在移动拷贝构造中，不需要加1。想一想现在是否就解决了shared_ptr的double free问题。
那么在intrusive_ptr_的析构函数中，他需要去判断refcount_是否为0，为0，则去调用target_的指定函数，并且free这个指针。不为0，则将自己的target置为空。 关键函数如下：

  void reset_() noexcept {
    if (target_ != NullType::singleton() &&
        detail::atomic_refcount_decrement(target_->refcount_) == 0) {
      // justification for const_cast: release_resources is basically a
      // destructor and a destructor always mutates the object, even for const
      // objects. NOLINTNEXTLINE(clang-analyzer-cplusplus.NewDelete)
      const_cast<std::remove_const_t<TTarget>*>(target_)->release_resources();

      // See comment above about weakcount. As long as refcount>0,
      // weakcount is one larger than the actual number of weak references.
      // So we need to decrement it here.
      if (target_->weakcount_.load(std::memory_order_acquire) == 1 ||
          detail::atomic_weakcount_decrement(target_->weakcount_) == 0) {
        delete target_;
      }
    }
    target_ = NullType::singleton();
  }

理解intrusive_ptr之后，我们就可以开始解析接下来tensor相关的代码了。

StorageImpl

c10/core/StorageImpl.h

struct C10_API StorageImpl final : public c10::intrusive_ptr_target {
 public:
  ....
 private:
  DataPtr data_ptr_;
  size_t size_bytes_;
  bool resizable_;
  // Identifies that Storage was received from another process and doesn't have
  // local to process cuda memory allocation
  bool received_cuda_;
  Allocator* allocator_;
};

StorageImpl定义如上，可以看到StorageImpl保存了一个DataPtr，用于表示当前这块tensor的buffer处于何种device上，并且内存地址是什么，Allocator对象用于传入不同device的内存分配器。这就是Tensor隐藏在背后的存储形式。 同时StorageImpl继承了c10::intrusive_ptr_target，说明了storageImpl可以由intrusive_ptr管理。 其诸多方法中我们可以看到它重载了父类的release_resources函数，用于其在refcount==0时释放资源。

void release_resources() override {
    data_ptr_.clear();
}

TensorImpl

c10/core/TensorImpl.h

TenorImpl已经和我们在Python层看到的Tensor产生了联系，TensorImpl的尾部有这样一个断言

static_assert(
    sizeof(void*) != sizeof(int64_t) || // if 64-bit...
        sizeof(TensorImpl) == sizeof(int64_t) * 24,
    "You changed the size of TensorImpl on 64-bit arch."
    "See Note [TensorImpl size constraints] on how to proceed.");

这个断言将TensorImpl结构体限制在了8*24字节，Pytorch代码中说这是因为在facebook的生产环境中会出现400百万的livetensor，这也导致了如果我们在TensorImpl中增加了一个64-bit的字段，都有可能让内存增加3.2G。当前TensorImpl中有如下成员

我们可以看到一个storage pointer，其他的都是为了保存当前Tensor的表示层信息strides，sizes，numel，以及一些自动微分，版本控制信息，还有一个PyObject pointer用于与Python层产生联系。
我们在Pytorch源码中搜索，其实还可以看到很多其他的TensorImpl，如SparseCsrTensorImpl，QTensorImpl。具体用处大家可以自行了解，但是我们可以以QTensorImpl为例，看一下如何扩展TensorImpl。

QtensorImpl继承与TensorImpl，但是它增加了一个字段 QuantizerPtr quantizer_;如何保证这个字段在以父类指针形式之间拷贝不丢失呢？

pytorch注明说，在他们希望两个variable共享相同的tensorImpl时，他们会调用shallow_copy_and_detach和shallow_copy_from这两个函数。

  // 1. `var_detached = var.detach()` uses `shallow_copy_and_detach()` to create
  // `var_detached` that shares the same tensor metadata with `var`, but with a
  // completely new autograd history.
  // 2. `var.set_data(tensor)` uses `shallow_copy_from()` to copy tensor
  // metadata from `tensor` into `var`, while keeping `var`'s original // AutogradMeta.

那么如果你需要实现一个自己的特定的XTensorImpl，在继承TensorImpl的同时，你需要重载父类的这个虚函数。这样在调用detach和set_data时你才能不丢失存储于XTensorImpl的特殊信息。
